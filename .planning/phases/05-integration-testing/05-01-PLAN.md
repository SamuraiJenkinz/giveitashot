  <name>Task 2: Write integration tests for dual-digest orchestration</name>
  <files>tests/test_integration_dual_digest.py</files>
  <action>
    Create `tests/test_integration_dual_digest.py` with the following test classes and methods. Use the fixtures from conftest.py (isolated_state, mock_emails_from_eml, load_eml, plus existing fixtures like message_center_major_update, regular_internal_email). All state tests MUST use `isolated_state` fixture (tmp_path) for isolation -- never touch the real .state.json.

    **Class 1: TestMultiRunStateSimulation**
    Test consecutive hourly runs with state persistence.

    - `test_state_transitions_across_5_runs`: Simulate 5 consecutive runs:
      1. Empty inbox (no emails) -- state should record regular_last_run, major_last_run stays None
      2. Regular only (3 regular emails) -- regular_last_run updated
      3. Mixed (2 regular + 2 major) -- both timestamps updated
      4. Major only (1 major email) -- major_last_run updated, regular_last_run unchanged from run 3
      5. Empty inbox again -- timestamps unchanged from run 4

      For each run: create a StateManager with the SAME state file (not new tmp_path per run -- reuse the file across runs to simulate persistence). Verify get_last_run returns correct value after each set_last_run. Use datetime.now(timezone.utc) + timedelta(hours=N) for timestamps to avoid hardcoded dates.

    - `test_state_persists_across_manager_instances`: Create StateManager, set regular last_run, destroy instance. Create NEW StateManager with same state file path. Verify get_last_run returns the previously set value. Proves file persistence works.

    - `test_independent_digest_type_state`: Set regular and major timestamps independently. Verify setting one does not affect the other. Clear one type, verify other is preserved.

    **Class 2: TestStateCorruptionRecovery**
    Test resilience to corrupted state files.

    - `test_completely_corrupted_json`: Write `"{ invalid json"` to state file. Create new StateManager. Verify both get_last_run("regular") and get_last_run("major") return None (clean reset). Verify no exception raised.

    - `test_empty_state_file`: Write empty string to state file. Create new StateManager. Verify clean reset, no exception.

    - `test_missing_state_file`: Create StateManager with nonexistent path. Verify clean state (both return None), no exception.

    - `test_invalid_timestamp_format`: Write valid JSON with `{"regular_last_run": "not-a-date", "major_last_run": "2026-02-26T10:00:00+00:00"}`. Create StateManager. Verify get_last_run("regular") returns None (corrupted field), get_last_run("major") returns the valid datetime. This tests per-field corruption isolation.

    - `test_wrong_schema_extra_fields`: Write valid JSON with extra unexpected fields plus valid last_run fields. Verify StateManager loads successfully, ignores extra fields, returns correct timestamps.

    - `test_truncated_json_file`: Write `{"regular_last_run": "2026-02-` (truncated). Create StateManager. Verify clean reset.

    **Class 3: TestFailureIsolation**
    Test that one digest type's failure doesn't affect the other.

    - `test_major_digest_exception_does_not_crash_program`: Mock the major digest path to raise an exception. Verify the main() function's try/except around major digest processing catches it (test the pattern from main.py lines 245-325). Verify the function continues execution after the major digest exception.

      Implementation approach: Don't call main() directly (requires full EWS auth). Instead, test the error isolation pattern by:
      1. Creating mock regular_emails and major_update_emails lists
      2. Verifying that a try/except block around MessageCenterExtractor().extract_batch() catches exceptions
      3. Asserting that the regular digest state is unaffected

    - `test_regular_digest_failure_preserves_major_state`: Set up isolated_state with major_last_run set. Simulate regular digest failure. Verify major_last_run is still intact after the failure.

    - `test_classification_failure_treats_all_as_regular`: Create EmailClassifier, mock classify to raise Exception. Verify that following the main.py pattern (try classify_batch, except treat all as regular), all emails end up in regular list and major list is empty.

    **Class 4: TestEdgeCases**
    Test boundary conditions for dual-digest system.

    - `test_empty_inbox_no_digest_generated`: With empty email list, verify no summarizer calls needed, system returns cleanly.

    - `test_all_emails_are_major_updates`: Pass list of only major update emails through classify_batch. Verify regular_emails is empty, major_update_emails contains all.

    - `test_all_emails_are_regular`: Pass list of only regular emails through classify_batch. Verify major_update_emails is empty.

    - `test_duplicate_mc_ids_deduplicated`: Use duplicate_mc_emails fixture. Run through MessageCenterExtractor.extract_batch then deduplicate. Verify only 1 result with is_updated=True.

    - `test_major_digest_disabled_skips_processing`: Test that when Config.is_major_digest_enabled() returns False (mock MAJOR_UPDATE_TO as empty), major update emails are classified but no major digest is generated (following main.py pattern).

    - `test_selective_execution_flags`: Verify the --regular-only and --major-only logic patterns: when regular_only=True, major digest code is skipped; when major_only=True, regular digest code is skipped.

    **Class 5: TestEndToEndPipeline**
    Test the full classification -> extraction -> formatting pipeline (without EWS/email sending).

    - `test_full_pipeline_mixed_emails`: Create 5 emails (3 regular + 2 major update using existing conftest fixtures). Run through:
      1. EmailClassifier.classify_batch -> get regular and major lists
      2. MessageCenterExtractor.extract_batch on major list -> get MajorUpdateFields
      3. MessageCenterExtractor.deduplicate -> get unique updates
      4. EmailSummarizer.format_major_updates_html(updates) -> get HTML string
      5. Assert HTML contains expected MC IDs, urgency sections, and is non-empty

    - `test_full_pipeline_synthetic_eml_files`: Load synthetic .eml fixtures via load_eml fixture, convert to Email objects via mock_emails_from_eml. Run same pipeline. Assert classification is correct for each synthetic email type.

    Use `from datetime import datetime, timezone, timedelta` for all time operations. Never hardcode dates that could break on different days. Use parametrize where appropriate to reduce boilerplate.
